# Contributing to Links & More

First off, thank you for considering contributing to Links & More! It's people like you that make Links & More such a great tool.

## How Can I Contribute?

### Reporting Bugs

This section guides you through submitting a bug report for Links & More. Following these guidelines helps maintainers and the community understand your report, reproduce the behavior, and find related bugs.

Before creating bug reports, please check existing issues as you might find out that you don't need to create one. When you are creating a bug report, please include as many details as possible. Fill out the **bug report template** to help us resolve the issue faster.

### Suggesting Enhancements

This section guides you through submitting an enhancement suggestion for Links & More, including completely new features and minor improvements to existing functionality.

Before creating enhancement suggestions, please check existing issues as you might find out that you don't need to create one. When you are creating an enhancement suggestion, please include as many details as possible and fill out the **feature request template**.

### Your First Code Contribution

Unsure where to begin contributing to Links & More? You can start by looking through `beginner` and `help-wanted` issues.

#### Local Development

1. Ensure you have the .NET 10 SDK installed.
2. Clone the repository: `git clone https://github.com/ian-cowley/LinksAndMore-`
3. Build the project: `dotnet build`
4. Run the app: `dotnet run` (from the project directory)

### Pull Requests

*   Fill in the **Pull Request template**.
*   Ensure the code follows the existing style of the project.
*   Update documentation if necessary.
*   Make sure all tests pass (if applicable).

## Styleguides

### Git Commit Messages

*   Use the present tense ("Add feature" not "Added feature")
*   Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
*   Limit the first line to 72 characters or less
*   Reference issues and pull requests liberally after the first line

### C# Styleguide

*   Follow standard C# naming conventions (PascalCase for classes/methods, camelCase for local variables).
*   Use explicit access modifiers (public, private, etc.).
*   Keep methods small and focused on a single responsibility.

## Code of Conduct

This project and everyone participating in it is governed by the [Links & More Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code.
